# extend-asset-sync-to-cover-local-claude-skills-and-hooks

## What's missing today

`scripts/sync_plugin_assets.py` already covers three sync directions:

| Target | Source | Sync trigger | Drift detection |
|---|---|---|---|
| `claude-plugin/skills/` | `goc/templates/skills/` (minus openclaw-only) | pre-commit | CI `--check` |
| `claude-plugin/goc/` | `goc/` (minus templates/skills) | pre-commit | CI `--check` |
| `openclaw-plugin/goc/` | `goc/` (minus templates/skills + hooks) | pre-commit | CI `--check` |

The dogfood self-host consumer-copy under `.claude/` has **no CI parity**:

| Target | Source | Sync trigger | Drift detection |
|---|---|---|---|
| `.claude/skills/<name>/SKILL.md` | `goc/templates/skills/<name>/SKILL.md` | `goc upgrade` (manual) | **none** |
| `.claude/hooks/<name>.py` | `goc/templates/hooks/<name>.py` | `goc upgrade` (manual) | **none** |
| `.claude/skills/_goc-bootstrap.sh` | `goc/templates/bootstrap/_goc-bootstrap.sh` | `goc install` (manual) | **none** |

`goc/engine.py:519`'s `validate_skill_dir_parity()` checks `.claude/skills/<name>/`
*existence* (run by `goc validate`, which is in pre-commit) but does NOT
compare file *contents*. If a maintainer edits
`goc/templates/skills/kickoff/SKILL.md` but forgets to mirror the change
to `.claude/skills/kickoff/SKILL.md`, the in-repo Claude Code session
keeps loading the stale body, with no warning.

## Why now

Empirically `.claude/skills/` and `goc/templates/skills/` are byte-identical
right now — the maintainer has been disciplined. But the very fact that
this conversation surfaced the question is the signal: a sync rule that
depends on disciplined memory will drift the first time someone is in a
hurry. The existing pre-commit + CI pattern for plugin payloads is the
natural home for the dogfood targets too.

## Design

Extend `scripts/sync_plugin_assets.py` with one new primitive
(`preserve_files`) and three new sync pairs.

### `preserve_files` parameter

A new `frozenset[str]` parameter on `_sync_dir`, distinct from `excludes`:

| Semantics | `excludes` | `preserve_files` |
|---|---|---|
| Source contains it | not copied to dst | source never has these — they exist only in dst |
| Destination contains it | actively deleted | left alone (never copied, never deleted) |
| Use case | `openclaw-*` skills must NOT ship to Claude plugin | `_goc-bootstrap.sh` is generated by install and must NOT be deleted by sync |

`_check_changes()` follows the same semantics: `preserve_files` paths
are excluded from the dst-side scan that flags "in dst but not in src".

### New sync pairs

1. `goc/templates/skills/` → `.claude/skills/`
   - excludes: openclaw-only skill dirs (same set computed for the
     Claude plugin payload)
   - preserve: `_goc-bootstrap.sh` (top-level file in `.claude/skills/`
     that's written by `goc install` from a different template path)
2. `goc/templates/bootstrap/_goc-bootstrap.sh` → `.claude/skills/_goc-bootstrap.sh`
   - single-file sync (the bootstrap source's canonical home, mirrored
     into `.claude/skills/` so the file is up-to-date with the engine)
3. For each hook name: `goc/templates/hooks/<name>.py` → `.claude/hooks/<name>.py`
   - same pattern as the Claude plugin's per-hook sync

### What stays manual

- `.claude/settings.json` — user/project-specific allow-list and
  hooks.json reference. Not synced.
- `AGENTS.md` and `CLAUDE.md` content above/below the marker block.
  The marker block itself is rewritten by the release workflow (already
  done in the previous card); the surrounding content is human-edited.
- `.game-of-cards/` content stubs and templates (e.g. project-specific
  README under `.game-of-cards/`).

## Out of scope

- Running `goc upgrade` from pre-commit. That command does too much
  (writes settings.json, the marker block, `.game-of-cards/` stubs) to
  be safe as an autonomous pre-commit step. The narrow file-pair sync
  is the right granularity.
- Detecting drift in *other* consumer-side repos. Each consumer is
  responsible for running `goc upgrade` when they want the latest
  templates; we are not in the business of policing remote repos.
- Folding `.claude/settings.json` into the sync. Repo-specific
  permissions live there and humans need to edit it.
