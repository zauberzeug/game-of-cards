# workflows-write-in-yaml-permissions-block-breaks-autonomous-workflows

## Why this is broken

The workflow-level `permissions:` block can only down-scope the
default `GITHUB_TOKEN` to scopes the `github-actions[bot]` App already
owns. The App's grant set (visible on any check-run via
`gh api /repos/.../check-runs`) covers `actions`, `attestations`,
`checks`, `contents`, `deployments`, `discussions`, `issues`,
`merge_queues`, `metadata`, `models`, `packages`, `pages`,
`pull-requests`, `repository_hooks`, `repository_projects`,
`security_events`, `statuses`, `vulnerability_alerts` — no
`workflows`. GitHub Actions' documented design explicitly prevents
the default `GITHUB_TOKEN` from editing files under
`.github/workflows/`, to block unauthorized escalation of bot
privileges. There is no `permissions:` key that overrides that.

Naming a scope GitHub doesn't recognise in `permissions:` fails the
workflow's parse step. The run ends as `conclusion: failure` with
`latest_check_runs_count: 0` and a 0s `run_started_at` ≈ `updated_at`
window — symptoms confirmed for runs 25634686279, 25634862771,
25635028873, 25635030737, 25635212085, 25635211981.

## Evidence

- Commit `34ddd96` introduced six lines (three per file) adding
  `workflows: write` with comments.
- Every `push` event to `main` since that commit produces a failed
  check-run for `audit-deck.yml` and `pull-card.yml` with zero jobs.
- The 02:10 UTC scheduled run of `audit-deck.yml` (run 25618417477)
  on 2026-05-10 — *before* commit `34ddd96` — succeeded normally.
- The `github-actions[bot]` App permissions object returned with any
  check-run's payload omits `workflows` entirely, so even if the YAML
  key were valid it could not be satisfied.

## Fix

Delete the `workflows: write` lines and their trailing comments from
both workflow files. That restores parseability and the workflows
resume on their `schedule` and `workflow_dispatch` triggers.

The original goal (let autonomous bot runs push edits under
`.github/workflows/`) cannot be reached by editing the YAML
permissions block — it requires routing a different token through
`claude-code-action`'s `github_token` input. Examples:

- a PAT with `workflow` scope, stored as a repo secret;
- a fine-grained PAT scoped to this repo with the "Workflows"
  permission granted;
- a GitHub App installation token from an App that holds the
  `workflows` permission, generated by an upstream step and passed
  into `claude-code-action` as `github_token`.

Picking among these is out of scope for this card — file a follow-up
if the bot-edits-workflow-files capability is actually needed. The
current cards in flight do not require it: every workflow-touching
fix in the last week was authored by a human commit.

Workflow-touching cards continue to hand off to a human (or to a
follow-up that wires a real workflow-scoped token).

## Out of scope

- Adding a workflow-scoped PAT or App-token plumbing to the
  autonomous workflows.
- Re-running the failed `audit-deck` / `pull-card` runs from the
  historic 0s-duration failures — those slots are gone; the next
  scheduled tick resumes the cadence after the fix.
