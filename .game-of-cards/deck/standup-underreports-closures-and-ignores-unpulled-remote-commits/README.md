# standup-underreports-closures-and-ignores-unpulled-remote-commits

## Location

- `goc/templates/skills/standup/SKILL.md:42-56` — Section 3, "Closed since yesterday" closure scan (source of truth).
- `goc/templates/skills/standup/SKILL.md:6-16` — Preflight + Context; no sync-state check.
- Auto-synced mirror: `.claude/skills/standup/SKILL.md` (regenerated by `scripts/sync_plugin_assets.py`; do not hand-edit).

## What's broken

**Defect 1 — the closure scan reasons from filesystem mtime, which is git-blind.** Section 3 ships this:

```bash
find .game-of-cards/deck -name "log.md" -newer .game-of-cards/deck -mmin -1440 2>/dev/null | ...
```

`find -newer X` is *strictly* greater-than X's mtime. Git does not preserve commit time as file mtime — a pull/merge/clone writes every touched file (and the `.game-of-cards/deck` directory itself) at one instant, giving them identical mtimes. When every `log.md` equals the reference directory's mtime, **nothing is "newer than" it**, so the scan returns ~zero and standup prints "Nothing closed in the last 24h."

This contradicts the skill's own intro, which claims (line 22) it:

> collapses `scan-deck` + `git log` + manual `log.md` reading into one read.

It never touches `git log`; the closure recency is derived entirely from mtime.

**Defect 2 — no sync-state check.** The Preflight and Context blocks never `git fetch` or compare HEAD to upstream. Run standup on a tree that is behind the remote and it silently reports a stale deck — with no warning that the view may be incomplete. This fails *quietly* regardless of how Defect 1 is implemented.

## Empirical evidence

Observed live in this repo on 2026-05-27 after a routine sync landed a large autonomous bug-fix run:

```
commits in last 24h:                              169
old find/-newer scan → closures reported:           0   ("Nothing closed in the last 24h")
log.md files with mtime in last 24h (no -newer):    77
log.md files NEWER than the deck dir:                1   (deck dir + all log.md share mtime 15:01:56)
engine closed_at within last 24h:                   62   (58 done + 3 disproved + 1 superseded)
```

The deck directory and every freshly-written `log.md` shared the exact mtime `May 27 15:01:56` (the git-write instant), which is what collapsed the `-newer` test to a single match. The authoritative count — cards whose engine-maintained `closed_at` lands inside the window — is **62**, and it correctly spans all three terminal statuses, including the one `superseded` card that a commit-message grep would miscategorize.

See `reproduce.py` for a deterministic proof that does not depend on this repo's live history.

## Why it matters

Standup *is* the "what happened since yesterday" view. Reporting zero closures the morning after 62 cards closed doesn't just omit detail — it asserts a falsehood that reads as "nothing happened," and it does so silently. A human reading it trusts a stale picture and may re-file or re-investigate already-closed work. Because the failure mode is "empty output," nothing flags that the scan is broken; it looks like a quiet day.

The root cause both defects share: standup infers "recent activity" from local filesystem facts (mtime; working-tree contents) instead of from the records that actually carry the truth — the engine's `closed_at` field and git's relationship to the remote.

## Fix

**Defect 1 — drive Section 3 from `closed_at`, not mtime.** Replace the `find` block with a read of the engine's structured closure record:

```bash
goc --json --status all 2>/dev/null | python3 -c "
import json, sys, datetime
cards = json.load(sys.stdin)
now = datetime.datetime.now(datetime.timezone.utc)
def parse(s):
    if not s: return None
    try: d = datetime.datetime.fromisoformat(s.replace('Z','+00:00'))
    except Exception:
        try: d = datetime.datetime.strptime(s[:10], '%Y-%m-%d')
        except Exception: return None
    return d.replace(tzinfo=datetime.timezone.utc) if d.tzinfo is None else d
recent = [c for c in cards if (d := parse(c.get('closed_at'))) and 0 <= (now - d).total_seconds() <= 86400]
recent.sort(key=lambda c: c['closed_at'])
for c in recent:
    print(f\"{c['title']}: {c['status']} — {(c.get('summary') or '')[:70]}\")
print(f\"({len(recent)} closed in last 24h)\" if recent else 'Nothing closed in the last 24h.')
" 2>/dev/null || true
```

This is convention-independent (no reliance on commit-message wording), git-independent for the *count*, precise (uses the timestamp the engine sets on close), and uniformly covers `done` / `disproved` / `superseded`.

**Defect 2 — add a sync-state preflight.** Add a Context `!` block (and a one-line note in Preflight prose) that fetches and warns when behind upstream, degrading gracefully:

```bash
git fetch --quiet 2>/dev/null; \
behind=$(git rev-list --count HEAD..@{u} 2>/dev/null || echo 0); \
[ "${behind:-0}" -gt 0 ] && echo "⚠️ Local is $behind commit(s) behind upstream — pull before trusting the deck view below." || true
```

`@{u}` resolves the current branch's upstream (no hardcoded `origin/main`); a missing upstream or offline fetch falls through to no warning rather than erroring.
