# `goc migrate-list-style` leaves the bulk rewrite uncommitted

## Location

- `goc/engine.py:2744-2748` — subparser definition; only `--dry-run`, no `--commit` / `--no-commit`.
- `goc/engine.py:4801-4840` — `_cmd_migrate_list_style`; writes via `readme.write_text(rewritten)` then prints "Done. Run `goc validate` to confirm." No `_git_auto_commit` call.

## What's broken

Every other state-mutation verb in the engine auto-commits by default
and exposes a `--commit` / `--no-commit` pair. `_cmd_advance` is
representative (`engine.py:4391-4409`):

```python
def _cmd_advance(args):
    ...
    _mutate_pair(title, advancer, "advanced_by", "advances", add=True)
    print(f"advance: {title}.advanced_by += {advancer}; {advancer}.advances += {title}")
    commit_policy = _commit_override(commit, no_commit)
    if auto_commit_enabled(commit_policy):
        if _git_auto_commit([DECK_DIR / title, DECK_DIR / advancer], f"deck: {advancer} advances {title}"):
            print("  committed")
```

`_cmd_migrate_list_style` (`engine.py:4801-4840`) shares the same
disk-write shape — read each README, re-parse the frontmatter, re-emit
via `emit_frontmatter`, write back — but skips the auto-commit tail
entirely:

```python
def _cmd_migrate_list_style(args):
    """Re-emit every card to convert advances/advanced_by to block-style lists."""
    dry_run = args.dry_run
    ...
    changed: list[str] = []
    for card_dir in sorted(DECK_DIR.iterdir()):
        readme = card_dir / "README.md"
        ...
        rewritten = emit_frontmatter(fm, body=body)
        if rewritten != original:
            changed.append(card_dir.name)
            if not dry_run:
                readme.write_text(rewritten)
    ...
    else:
        print(f"Rewrote {len(changed)} card(s):")
        for name in changed:
            print(f"  {name}")
        print("Done. Run `goc validate` to confirm.")
```

The subparser at `engine.py:2744-2748` defines only `--dry-run`:

```python
p_mls = subparsers.add_parser("migrate-list-style",
                              help="Re-emit every card to convert advances/advanced_by to block-style lists.")
p_mls.add_argument("--dry-run", action="store_true",
                   help="Show which cards would change without writing files.")
```

No `--commit`, no `--no-commit`, no `_git_auto_commit`. After a real
(non-dry) run the entire deck rewrite sits in the working tree, dirty.

## Why it matters

`migrate-list-style` is a **bulk** mutator — it re-emits every card
whose serialized frontmatter differs from canonical, which can be the
entire deck on a list-style migration day. That makes the missing
auto-commit the highest-blast-radius instance of an already-documented
family:

- [goc-new-with-advances-leaves-parent-card-mutation-uncommitted](../goc-new-with-advances-leaves-parent-card-mutation-uncommitted/)
- [goc-move-leaves-cross-reference-rewrites-uncommitted](../goc-move-leaves-cross-reference-rewrites-uncommitted/)
- [goc-repair-edges-apply-leaves-edge-repairs-uncommitted](../goc-repair-edges-apply-leaves-edge-repairs-uncommitted/)

The reachability path is direct: any user (or release script) that
runs `uv run goc migrate-list-style` in a repo with drifted edge
fields produces a many-file working-tree diff and zero new commits.
`AGENTS.md` "Parallel-Agent Commit Safety" explicitly warns that the
git index is shared state across agents — a bulk uncommitted diff is
exactly the dangerous state two parallel sessions can collide on, and
this verb is a known producer.

The card-schema invariant the family violates: "verbs that mutate
deck files auto-commit by default; the `--commit` / `--no-commit` pair
overrides." Either the invariant holds for `migrate-list-style` too,
or it needs a documented carve-out.

## Decision required

The fix shape isn't predetermined. Two credible paths:

1. **Match the family (recommended).** Add `--commit` / `--no-commit`
   to `p_mls`, call `_git_auto_commit(changed_dirs, "deck:
   migrate-list-style: rewrite N cards to block-style")` after the
   write loop, gated by `_commit_override` like every sibling.
   Pro: preserves the invariant; the bulk rewrite becomes one
   reviewable commit; parallel-agent commit safety holds. Con: a
   single commit touching dozens of cards is large; reviewers may
   prefer per-card commits.
2. **Document as an intentional bulk-edit verb.** Print an explicit
   tail ("Bulk rewrite is intentionally not auto-committed. Now run:
   `git add .game-of-cards/deck && git commit -m '…'`") and add a
   prominent doc note. Pro: makes the human the gatekeeper of a
   large diff. Con: breaks the family invariant; the silent default
   today is the failure mode the doc would be papering over.

Sub-question for path 1: one commit for the whole rewrite, or one
commit per card? The repair-edges card landed on one-commit-per-pair;
that's the precedent. A bulk verb plausibly wants one summary commit.

## Fix sketch (if path 1 is chosen)

```python
# engine.py around 2744
p_mls.add_argument("--commit", action="store_true",
                   help="Force auto-commit for this bulk rewrite.")
p_mls.add_argument("--no-commit", action="store_true",
                   help="Skip auto-commit for this bulk rewrite.")

# engine.py around 4836 (after the print loop)
commit_policy = _commit_override(args.commit, args.no_commit)
if auto_commit_enabled(commit_policy):
    changed_dirs = [DECK_DIR / name for name in changed]
    if _git_auto_commit(changed_dirs, f"deck: migrate-list-style: rewrite {len(changed)} card(s) to block-style"):
        print("  committed")
```
