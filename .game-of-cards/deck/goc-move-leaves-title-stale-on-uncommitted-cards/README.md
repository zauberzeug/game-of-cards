# goc-move-leaves-title-stale-on-uncommitted-cards

## Location

- `goc/engine.py` — `_move_iter_tracked_text_files()` enumerates files with
  `git ls-files -z` (tracked only):

  ```python
  result = subprocess.run(
      ["git", "ls-files", "-z"],
      cwd=str(REPO_ROOT), capture_output=True, check=True, timeout=30,
  )
  ```

  The `except` fallback (only on `CalledProcessError` / `FileNotFoundError` /
  `TimeoutExpired` — i.e. *no git at all*) uses `rglob("*")`, which DOES see
  untracked files. So the bug only fires inside a real git repo.
- `goc/engine.py` — `_cmd_move()` calls `git mv` (which refuses an untracked
  source and falls through to `shutil.move`), then `_move_rewrite_tracked_files`
  (which iterates the tracked-only set above), then appends the renamed-from
  log entry.

## What's broken

`goc move` performs two actions that must agree: it renames the card
*directory*, and it rewrites the slug *inside* tracked text files —
including the moved card's own `title:`, `advances`, `advanced_by`, H1, and
cross-links (the function's own docstring promises "frontmatter
title/advances/advanced_by fields").

When the moved card was **never committed** — the extremely common flow of
`goc new <slug>` → notice the slug is wrong → `goc move <slug> <better-slug>`
before the first commit — its README.md and log.md are *untracked*.
`git ls-files` does not list them, so the in-file rewrite skips them
entirely. Meanwhile `git mv` refuses the untracked source and the code falls
back to `shutil.move`, which still renames the directory. The two halves
diverge: the directory is `NEW`, the `title:` field still says `OLD`.

`validate_card` then fails on the divergence:

```
ERROR: <OLD>: title: '<OLD>' != dir name '<NEW>'
```

The card is left self-contradictory and uncloseable until a human hand-edits
the `title:` field.

## Empirical evidence

`uv run python deck/goc-move-leaves-title-stale-on-uncommitted-cards/reproduce.py`:

```
dir renamed to card-with-typo-in-slug: True
in-file title line: 'title: card-with-typo-in-slog'
validate output (title-mismatch lines):
  ERROR: card-with-typo-in-slog: title: 'card-with-typo-in-slog' != dir name 'card-with-typo-in-slug'

FAIL: move renamed the directory but left title: stale → card fails validate
```

A *committed* card moves correctly (the README is tracked, so `git ls-files`
lists it and the rewrite runs) — confirming the defect is specifically the
untracked case.

## Why it matters

Reachability path: every card filed with `goc new` starts untracked. The
file-then-rename-before-commit flow is routine for agents (create card,
realize the slug reads poorly, fix it immediately). The move silently
produces a card that `goc validate` — the CI gate that protects against
frontmatter drift — rejects, with no warning at move time. This is the
opposite failure from
[move-rewrites-card-slug-inside-urls-paths-and-code-identifiers](../move-rewrites-card-slug-inside-urls-paths-and-code-identifiers/)
(move too *aggressive*); here move is too *narrow*. It is also distinct from
[goc-move-leaves-cross-reference-rewrites-uncommitted](../goc-move-leaves-cross-reference-rewrites-uncommitted/),
which is about move not *auto-committing* its (correct) rewrites — that card
assumes the rewrites happened; this one is about them never happening at all.

## Fix

Broaden `_move_iter_tracked_text_files` to enumerate untracked-but-not-ignored
files alongside tracked ones, matching the `rglob` fallback's intent (see
everything except `.git`/ignored):

```python
result = subprocess.run(
    ["git", "ls-files", "-z", "--cached", "--others", "--exclude-standard"],
    cwd=str(REPO_ROOT), capture_output=True, check=True, timeout=30,
)
```

`--cached` keeps tracked files, `--others --exclude-standard` adds untracked
files that are not gitignored. Build the path list through an order-preserving
dedupe (a tracked + modified file appears once anyway, but guard defensively).
Ignored artifacts (venv, build output) stay excluded, so the broadened sweep
does not start rewriting unrelated trees.
